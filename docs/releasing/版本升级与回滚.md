# 版本升级与回滚

## 1. 三类版本

UEKit Web 同时存在：

- CLI npm 版本，如 `@uekits/web@0.2.0`。
- Registry 条目版本，如 `pro-table@0.3.0`。
- Registry 协议版本，如 `/web/v1` 和 `schemaVersion: 1`。

三者不要混淆。升级 CLI 不代表自动升级本地组件。

## 2. CLI 升级

使用 `pnpm dlx` 时可以显式指定版本：

```bash
pnpm dlx @uekits/web@0.2.0 info
```

先阅读变更说明，再在升级分支运行。CLI 重大版本可能涉及配置或 Lock 迁移。

## 3. Registry 条目升级

```bash
pnpm dlx @uekits/web@0.2.0 diff pro-table
pnpm dlx @uekits/web@0.2.0 update pro-table
```

本地有修改时：

```bash
pnpm dlx @uekits/web@0.2.0 update pro-table --merge
```

按[源码安装与升级机制](../architecture/源码安装与升级机制.md)处理冲突。

## 4. 协议升级

Registry 协议重大升级应使用新路径，例如 `/web/v2`，并在过渡期保留 v1。CLI 需要：

- 识别旧配置。
- 提供明确迁移提示。
- 避免把旧 Lock 误当新协议。
- 在 CI 同时验证受支持版本。

## 5. 业务项目升级批次

推荐：

1. Playground 验证。
2. 内部样板项目验证。
3. 一个低风险真实项目试点。
4. 租户端和平台端各验证一条完整链路。
5. 再推广到其他项目。

## 6. 回滚单位

一次升级提交应包含：

- 本地组件源码。
- `uekit.lock.json`。
- `.uekit/bases`。
- `package.json` 和包管理器锁文件。
- 必要的业务适配。

这样可以使用 Git 整体回滚。

## 7. Registry 回滚

Revert Registry 源提交并重新部署。不要直接在线手改生成 JSON。

## 8. CLI 回滚

npm 不能覆盖历史版本。通过修复版本、deprecated 和 dist-tag 管理。业务项目可以临时固定上一稳定 CLI：

```bash
pnpm dlx @uekits/web@0.1.0 add button
```

## 9. 数据兼容

UEKit Web 当前主要修改源码和配置，不迁移业务数据库。但 `uekit.json`、Lock 和 Base 本身也是状态数据，任何不兼容升级都必须有迁移和回滚设计。

## 10. 紧急故障处理

```text
停止推广升级
  → 固定上一稳定 CLI/Registry 条目
  → Revert 或发布修复版本
  → 完成最小消费项目和关键真实页面验证
  → 恢复发布
  → 补充原因和防回归测试
```

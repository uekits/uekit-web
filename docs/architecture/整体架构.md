# 整体架构

## 1. 架构目标

UEKit Web 需要同时满足：

- 组件可以按需安装，而不是全量引入。
- 消费方拥有源码并可以修改。
- 修改后的源码仍能追踪上游差异和升级。
- Registry 能部署为纯静态资源。
- CLI 与 Registry 可以独立发布。
- 组件在进入真实项目之前已经经过浏览器和发布物验证。

```text
│                                      └─► GitHub Pages   │
│                                                        │
│  packages/cli ─────────► npm @uekits/web               │
│                                                        │
│  apps/playground ◄──── Registry 标准源码               │
│        │                                               │
│        └─► Vitest / Playwright                          │
└────────────────────────────────────────────────────────┘

                       网络或本地路径
                              │
                              ▼
┌────────────────────── 消费方 Vue 项目 ──────────────────┐

│   └─ src 中的本地组件、布局、主题和集成源码             │

```

## 3. 物理结构

```text
uekit-web/
├─ apps/
│  ├─ playground/          # 组件展示和端到端测试
│  └─ registry-server/     # 静态 Registry 构建壳
├─ packages/
│  └─ cli/                 # @uekits/web
├─ registry/               # 标准组件源码
├─ registry-releases/      # 已公开条目的不可变版本档案
├─ registry.json           # 条目清单
├─ registry.schema.json    # 条目协议 Schema
├─ uekit.schema.json       # 消费项目配置 Schema
├─ scripts/                # 构建产物与发布物验证
├─ tests/                  # 单元和 E2E 测试
└─ docs/                   # 中文文档
```

## 4. 三条核心链路

### 4.1 组件生产链路

```text
识别通用能力
  → 编写 registry 源码
  → 登记 registry.json
  → 提升条目版本并归档 registry-releases
  → Registry 构建校验
  → Playground 展示
  → Vitest / Playwright
  → 发布 Registry
```

### 4.2 组件消费链路

```text
CLI 获取 Registry
  → 解析 Registry 依赖
  → 安装 npm 依赖
  → 改写源码别名
  → 写入消费项目目录
  → 记录 Lock 与 Base
  → 业务页面引用本地源码
```

### 4.3 组件升级链路

```text
读取 Lock
  → 比较 Local 与 Base
  → 获取 Remote 新版本
  → 未修改则覆盖 / 已修改则保护
  → 可选三方合并
  → 处理冲突
  → 测试并提交
```

## 5. 发布边界

### npm 发布物

`@uekits/web` 只包含 CLI 执行所需代码，不承载消费项目组件运行时。

### Registry 发布物

`registry.uekit.com/web/v1` 提供索引、条目 JSON 和公开 Schema。Registry 是静态资源，可以由 GitHub Pages、CDN 或企业私有静态服务托管。

### 消费方发布物

业务应用构建时，组件已经是消费项目源码。线上运行不依赖 UEKit Registry，也不需要浏览器动态下载 UEKit 组件。

## 6. 关键边界

- CLI 不保存业务数据。
- Registry 不执行消费项目代码。
- Playground 不承载行业业务实现。
- Integration 可以封装第三方 SDK，但密钥必须由消费项目环境提供。
- UEKit 源码安装完成后，运行时稳定性由消费项目构建产物保证。

## 7. 为什么使用 Monorepo

CLI、Registry、Playground 和验证脚本需要原子协作：

- 修改条目协议时同时修改 CLI 和构建器。
- 修改组件时立即在 Playground 验证。
- 发布前可用同一提交构建 CLI tarball 和 Registry。
- CI 能验证两类发布物的兼容关系。

Monorepo 是源码协作方式，不代表所有内容要发布成多个 npm 包。

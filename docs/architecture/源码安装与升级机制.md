# 源码安装与升级机制

## 1. 为什么需要升级机制

源码复制到本地后，调用方可以自由修改，但上游也会继续修复缺陷和增加能力。如果没有版本、哈希和基准，后续只能靠肉眼比较两个目录。

UEKit 使用三份状态建立可追踪关系：

```text
Remote：Registry 当前版本
Base：安装时保存在 .uekit/bases 的上游源码
Local：调用方当前使用的源码
```

## 2. 安装状态

| 状态           | 判断                    | 默认行为         |
| -------------- | ----------------------- | ---------------- |
| 未安装         | Lock 中不存在           | 写入             |
| 已安装未修改   | Local hash = Lock hash  | 允许更新         |
| 已安装已修改   | Local hash != Lock hash | 保护并跳过       |
| 未追踪同名文件 | 文件存在但 Lock 无记录  | 拒绝覆盖         |
| 本地文件丢失   | Lock 有记录但文件不存在 | 重新安装前需确认 |

`--overwrite` 是显式放弃保护，不应作为日常默认参数。

## 3. Base 的作用

安装条目时，CLI 把相同版本源码写入：

```text
.uekit/bases/<item>/<version>/<relative-file>
```

Base 用于判断调用方相对上游做了什么修改、展示差异并为三方合并提供共同祖先。删除 Base 后，Lock 仍知道文件哈希，但无法可靠执行内容级三方合并。

## 4. diff

```bash
pnpm dlx @uekits/web@latest diff pro-table
```

比较的是 Local 与安装时 Base，而不是直接与最新 Remote 比较。这样结果准确表达“团队改过什么”。

## 5. 普通更新

```bash
pnpm dlx @uekits/web@latest update pro-table
```

如果 Local 未修改，CLI 获取 Remote 并覆盖源码、更新 Base 和 Lock。如果 Local 已修改，默认不覆盖。

## 6. 三方合并

```bash
pnpm dlx @uekits/web@latest update pro-table --merge
```

概念上执行：

```text
merge(Local, Base, Remote)
```

- 无冲突：得到同时包含本地修改和上游变更的新 Local。
- 有冲突：保留本地正式文件，另存 `.uekit-merge` 供人工处理。

两种结果都会把 Lock 版本和 Base 推进到新的 Remote 基线。无冲突时，后续 `diff` 仍只展示 Local 相对新 Base 保留的定制；有冲突时，Local 保持不变，`.uekit-merge` 保存候选合并内容。

## 7. 当前实现边界

当前三方合并属于早期能力：

- 文件对应关系使用别名展开后的完整目标路径，不按 basename 猜测。
- 上游重命名、拆分或合并文件时，旧文件会被安全保留，不会自动删除或迁移本地定制。
- 缺少旧 Base 时不会尝试伪合并，而是保留 Local 并生成 `.uekit-merge`。
- 多文件升级还不是事务操作，中途发生文件系统或依赖安装错误时需要通过 Git 回滚。
- 不能替代业务测试和人工代码评审。

因此 `--merge` 的正确定位是“减少机械合并工作”，不是“一键安全升级”。

## 8. 推荐升级流程

```bash
git switch -c chore/upgrade-uekit
pnpm dlx @uekits/web@0.2.0 info
pnpm dlx @uekits/web@0.2.0 diff pro-table
pnpm dlx @uekits/web@0.2.0 update pro-table --merge
pnpm typecheck
pnpm test
pnpm build
```

随后处理 `.uekit-merge`，检查 Lock 与 Base，并用 Playwright 验证关键页面。

## 9. 回滚

最可靠的回滚方式是 Git：

```bash
git revert <upgrade-commit>
```

一次规范升级提交应同时包含组件、依赖、Lock 和 Base。不要只复制回旧 Vue 文件而保留新 Lock 和新 Base。

## 10. 规划中的增强

- 升级计划预览。
- 基于完整路径和内容相似度的重命名识别。
- 冲突解决后的 Lock/Base 原子提交。
- Registry Migration 和 Codemod。
- 批量更新时的事务性回滚。
- 可验证来源和签名。

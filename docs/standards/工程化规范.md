# 工程化规范

本文定义 UEKit Web 的开发环境、工具职责、质量门禁和源码分发流程。目标是在团队风格一致的前提下，保留 Monorepo、Node CLI 和 Registry 源码分发所需的架构差异。

## 1. 规则等级

- **MUST**：开发、CI 和发布流程必须满足。
- **SHOULD**：默认采用；偏离时需要记录技术理由。
- **MAY**：按模块需要选用，不强制全仓统一启用。

## 2. 仓库模型

UEKit Web 不是普通单体 Vue 业务项目，而是同时维护以下交付物：

```text
registry/                  可分发源码，是组件实现的唯一源
packages/cli/              @uekits/web Node CLI
apps/playground/           Registry 的真实 Vue 消费和验收应用
apps/registry-server/      Registry 静态站点构建入口
scripts/                   仓库、Registry 和 npm 包校验脚本
tests/                     单元、消费夹具和端到端测试
```

- `registry/` MUST 是可分发组件的唯一人工维护源。
- `registry/package.json` 仅作为私有类型验证 Workspace，MUST 保持 `private: true`，不得作为组件运行时包发布。
- Playground 中通过 CLI 安装的组件 MUST 由 Registry 同步，不得与 Registry 同时手工修改。
- `dist/`、Registry 构建 JSON、测试报告和 `.uekit/bases` 等生成内容不得作为源文件直接修复。
- 根配置只维护全仓共同规则，浏览器和 Node 差异在各自配置层表达。

## 3. 团队规范适配矩阵

| 主题               | 团队业务工程基线                   | UEKit Web 决策                                         | 类型     |
| ------------------ | ---------------------------------- | ------------------------------------------------------ | -------- |
| 缩进和引号         | 4 空格、单引号、分号               | 完全采用                                               | 直接继承 |
| Vue SFC 顺序       | `template`、`script`、`style`      | 完全采用，文件头置于最前                               | 直接继承 |
| 文件头和 TSDoc     | 正式源码文件头、公共 API TSDoc     | 完全采用，并强调 Registry 的消费方可读性               | 直接继承 |
| 包管理器           | Yarn 4                             | 使用 pnpm Workspace、Catalog 和 Corepack               | 架构例外 |
| Node 版本          | `.nvmrc`、`.node-version` 固定版本 | 采用版本文件，但与根 `engines` 和 CI 统一              | 适配继承 |
| TypeScript         | 浏览器应用配置                     | 拆分公共、Node CLI、Vue 浏览器配置                     | 架构适配 |
| Element Plus       | 组件显式导入、插件自动导入样式     | Registry 显式导入组件和样式，不假设消费方插件          | 架构例外 |
| Tailwind           | 业务项目配置文件                   | 保持 Tailwind v4 CSS-first；只有真实需求才增加配置文件 | 架构例外 |
| Vitest             | jsdom 应用测试                     | CLI/Registry 使用 Node；Vue DOM 测试使用 happy-dom      | 架构适配 |
| `console`          | 浏览器生产源码限制                 | 浏览器限制；CLI 的 stdout/stderr 是公开交互            | 目录例外 |
| Router、Pinia、API | 业务应用专项规范                   | 不进入通用 Registry/CLI 规范                           | 不适用   |
| Video/AIOT         | 参考项目业务专项                   | 不复制；未来按 Integration 独立制定                    | 不适用   |
| Husky/lint-staged  | Yarn 命令                          | 使用 `corepack pnpm lint-staged`                       | 命令适配 |
| 自动导入           | 禁止组件全局自动导入               | 继续禁止；配置工具的静态插件不等于组件隐式依赖         | 直接继承 |

“架构例外”不是降低标准。例外必须在配置中限定目录、运行环境和原因，不得通过全局关闭规则实现。

## 4. 运行环境和包管理

- 仓库 MUST 使用根 `package.json` 声明的 Node 和 pnpm 基线。
- 开发机和 CI MUST 通过 Corepack 使用声明的 pnpm 版本。
- MUST 提供内容一致的 `.nvmrc` 和 `.node-version`，精确版本应落在 `engines.node` 范围内。
- MUST 只提交 `pnpm-lock.yaml`；禁止在仓库中产生 npm、Yarn 或 Bun 锁文件。
- MUST 使用 `pnpm-workspace.yaml` 维护 Workspace 和 Catalog 版本。
- CI MUST 使用冻结锁文件安装，禁止静默更新依赖解析结果。
- 根脚本 SHOULD 使用 `corepack pnpm`，确保直接执行和 CI 行为一致。

Registry CLI 支持在消费项目识别 pnpm、npm、Yarn 和 Bun，这是产品能力，不改变 UEKit 仓库自身只使用 pnpm 的规则。

根仓库开发环境与发布物运行环境是两个基线：`.nvmrc` 和 `.node-version` 跟随根 `engines.node` 固定开发版本；`packages/cli` 和 Registry 清单中的 Node 范围描述消费方兼容性。调整开发版本不得顺带抬高 CLI 的最低运行版本，CLI 也不得使用超出其 `engines.node` 的 Node API。

## 5. 工具职责

### 5.1 EditorConfig 和 VS Code

- EditorConfig 负责字符集、换行、缩进和末尾空白等编辑器基础行为。
- VS Code 工作区配置负责固定格式化器、保存时 ESLint 修复、本地 TypeScript SDK 和文件排除。
- 推荐扩展 MUST 包含 Vue 官方扩展、ESLint、Prettier、Tailwind CSS 和 Playwright。
- 工作区设置不得依赖个人全局配置才能正确格式化。

### 5.2 Prettier

- Prettier 只负责确定性格式，不负责代码正确性。
- 配置目标为单引号、分号、4 空格、100 字符、尾逗号和 Vue 多属性换行。
- Markdown SHOULD 保留人工段落布局；构建产物、锁文件和第三方源码 MUST 排除。
- 不得用 ESLint 重复实现与 Prettier 冲突的格式规则。

### 5.3 ESLint 和 TSDoc

- ESLint 负责 JavaScript、TypeScript、Vue 和 TSDoc 的静态质量规则。
- 浏览器、Node、测试和配置文件 MUST 使用分域 globals，不得全仓同时开放 Browser 和 Node。
- 规则禁用 MUST 限定最小范围并说明原因。
- TSDoc 插件负责语法，文件头脚本负责存在性，语义质量由代码审查负责。

### 5.4 TypeScript

TypeScript 配置 SHOULD 采用分层结构：

```text
tsconfig.base.json               与环境无关的严格规则
tsconfig.json                    根项目引用或入口
registry/tsconfig.json           Registry 事实源的 Vue SFC 与公开类型
packages/cli/tsconfig.json       NodeNext、Node 类型、CLI 构建输出
apps/playground/tsconfig.json    DOM、Vite 和 Vue SFC
```

- 公共配置不得无条件包含 DOM 和 Node 两套类型。
- Registry MUST 通过私有 Workspace 声明所有条目的类型验证依赖，并由 `vue-tsc` 直接检查事实源；不得用 Playground 副本替代。
- Registry 的 `paths` MUST 显式表达标准消费别名到事实源目录的映射，包括 `foundation/icon` 与最终 `{ui}/icon` 的目录差异。
- `noUncheckedIndexedAccess` 等新增严格规则 MUST 分批启用并真实修复问题，不得用断言压制。
- 构建配置和应用源码可使用不同的 `include`、`types`、module 和 emit 策略。

### 5.5 Stylelint

- Stylelint 负责 CSS、SCSS 和 Vue style 块。
- UEKit Token、Tailwind v4 指令和 Vue `:deep()` 例外 MUST 精确配置。
- 不得关闭能够发现真实全局污染、无效属性或选择器错误的规则。

## 6. Git Hook 和增量检查

- 仓库 MUST 使用 Husky 管理 `pre-commit`。
- `pre-commit` MUST 先执行仓库安全扫描，再通过 `corepack pnpm lint-staged` 检查暂存文件，不调用 Yarn。
- lint-staged 只处理暂存文件，适合运行 Prettier、ESLint 和 Stylelint 自动修复。
- TypeScript 全量检查、单元测试、构建和 E2E SHOULD 留给 `pnpm check` 或 CI，避免提交钩子不可接受地缓慢。
- Hook 不替代 CI；开发者不得通过长期 `--no-verify` 绕过失败。

## 7. 质量门禁

### 7.1 本地静态检查

```sh
pnpm format:check
pnpm security:check
pnpm docs:verify
pnpm check:file-headers
pnpm lint
pnpm lint:styles
pnpm typecheck
pnpm test
```

安全扫描和文件头检查 MUST 纳入根 `check`。

### 7.2 构建和发布物

```sh
pnpm registry:build
pnpm registry:verify
pnpm package:check
pnpm test:e2e
pnpm smoke:package
```

- `pnpm check` MUST 保持全仓确定性验证入口。
- E2E 和 tarball 冒烟 MAY 独立运行，但发布前 MUST 执行。
- 校验失败时修复源文件或协议，不得直接编辑生成产物。

## 8. Registry 变更流程

任何 Registry 源码修改，包括纯格式和注释，都会改变源码内容 Hash。标准流程是：

```text
修改 registry/ 源码
        ↓
按 SemVer 提升条目版本
        ↓
执行 registry:release 写入不可变归档
        ↓
执行 Registry 构建和验证
        ↓
通过 CLI 同步 Playground 并更新 Lock/Base
        ↓
执行类型、构建、E2E 和消费项目冒烟
```

- MUST 先修改 `registry/`，再生成发布物和 Playground 副本。
- 已发布条目的内容发生变化后 MUST 提升版本并执行 `pnpm registry:release`；不得删除或覆盖 `registry-releases` 中已经公开的版本。
- MUST 检查 `uekit.lock.json`、`.uekit/bases` 和 Registry Hash 的一致性。
- 已发布条目的源码内容变化 MUST 按 SemVer 更新条目版本；是否影响运行行为决定变更级别，而不是是否“只是格式”。
- 未正式发布的整体规范迁移 MAY 定义为一次初始基线，但必须在发布记录中说明。
- 不得同时手工编辑 Registry 与 Playground 的同名组件。

## 9. 配置变更流程

调整工程配置时 MUST：

1. 说明要约束的问题和适用目录；
2. 验证工具版本支持所用配置格式；
3. 先运行目标配置的最小检查；
4. 记录新增诊断并分批修复；
5. 更新开发文档和 CI 命令；
6. 确认没有把 Node 规则误用到浏览器或反向误用；
7. 提交锁文件变化和依赖许可证变化。

不得从参考项目复制未使用的配置，例如无实际需求的 Tailwind 配置、业务 Router、自动导入或 jsdom 全局设置。

## 10. 改动分层

规范迁移和大规模治理 MUST 按以下层次拆分：

1. 文档和规则决策；
2. 工具配置和依赖；
3. 机械格式迁移；
4. 注释、类型和行为治理；
5. Registry 重建和全链路验收。

纯格式提交不得夹带功能修改。类型规则暴露的真实问题必须以可审查的代码修改解决，并补充相应测试。

## 11. 生成物和忽略范围

以下内容 MUST 从格式、Lint 或文件头检查中按工具需要排除：

- `node_modules/`；
- 各级 `dist/`；
- `coverage/`；
- `playwright-report/`、`test-results/`；
- Registry 构建 JSON；
- 第三方 Vendor 源码；
- 包管理器锁文件的格式化处理。

排除生成物不等于跳过发布物验证。Registry、npm tarball 和消费项目必须通过专门的结构及行为检查。

## 12. 基线和工作区保护

- 大规模迁移前 MUST 记录 `git status` 和现有质量门禁结果。
- 工作区存在用户未提交改动时 MUST 保留，禁止使用 `git reset --hard`、覆盖式 checkout 等破坏性操作。
- SHOULD 先建立可恢复的 Git 基线，再执行全仓格式化；是否提交由仓库负责人决定。
- 每个迁移阶段完成后 MUST 复核改动范围，防止生成产物或无关文件进入变更集。

## 13. 当前配置基线

仓库已实施以下配置；后续调整必须同步维护脚本、CI 和本规范：

- `.editorconfig`：4 空格和通用文件规则；
- Prettier：使用可注释配置，包含团队格式选项和明确忽略范围；
- ESLint：TSDoc、类型导入、`any`、Vue 命名及分域环境；
- TypeScript：已拆分公共、Node CLI 和 Vue 浏览器配置，并启用严格规则；
- `.vscode/`：推荐扩展、保存格式化、ESLint 修复和本地 TypeScript；
- `.nvmrc`、`.node-version`：与 `engines`、CI 一致；
- Husky、lint-staged：使用 pnpm 的增量提交检查；
- 文件头检查脚本：排除 JSON、生成物和第三方文件；
- Stylelint：包含 Tailwind v4、Token 和 Vue 深度选择器例外；
- Vitest：CLI/Registry 默认使用 Node，Vue DOM 组件测试使用文件级 happy-dom 环境；
- Playwright：只包含当前仓库真实需要的网络和报告配置；
- 根脚本：文件头和安全扫描已接入 `check`，E2E 和 tarball 冒烟保持为发布前独立验收。

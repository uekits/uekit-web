# 依赖管理规范

仓库自身统一使用 pnpm、Corepack、Workspace 和 Catalog。CLI 对 pnpm、npm、Yarn、Bun 的识别属于消费项目兼容能力，不代表 UEKit 仓库可以混用包管理器。

## 1. 依赖分类

### CLI 依赖

只放 `@uekits/web` 命令运行所需依赖，例如 Commander。不要把 Element Plus、Vue、地图和图表 SDK 放入 CLI 运行时依赖。

### Registry 运行时依赖

写在条目的 `dependencies`，由 CLI 安装到消费项目。

### Registry 开发依赖

写在 `devDependencies`，例如 Tailwind 和 Vite 插件。

### Registry 条目依赖

写在 `registryDependencies`，不能伪装成 npm 包。

## 2. 版本范围

- 使用明确最低兼容版本。
- 避免 `*` 和无上界的重大版本。
- 底层框架重大版本升级必须有兼容测试。
- catalog 统一 Monorepo 内相同依赖版本。

## 3. 按需原则

- 安装地图才加入 AMap Loader。
- 安装图表才加入 ECharts。
- 不因 Playground 使用某依赖就把它加入所有条目。
- Element Plus 组件和样式按需导入。
- Registry 源码显式导入 Element Plus 样式，不依赖消费项目的自动样式导入插件。

## 4. 新增依赖评审

评估：

- 是否可用平台或已有依赖实现。
- 许可证是否与 MIT 项目兼容。
- 维护活跃度和安全记录。
- ESM、tree-shaking 和类型声明。
- 浏览器体积和运行开销。
- SSR 或 CSP 影响。

第三方许可证变化需要复核 `docs/legal/README.md`，并通过 `pnpm licenses:generate` 更新 `docs/legal/THIRD_PARTY_LICENSES.md`。

## 5. 锁文件

- 根 `pnpm-lock.yaml` 必须提交。
- 消费项目 `uekit.lock.json` 必须提交。
- CI 使用 `--frozen-lockfile`。
- 不手工编辑 pnpm lock。

## 6. 安全升级

依赖升级步骤：

1. 查看 Release Notes 和重大变更。
2. 更新 catalog 或条目规格。
3. 重新构建 Registry。
4. 运行 `pnpm check`、E2E 和 tarball 冒烟测试。
5. 在至少一个干净消费项目验证。
6. 对公开变化添加 Changeset。

## 7. 禁止事项

- 在源码中依赖未声明 npm 包。
- 依赖消费项目恰好全局注册的组件。
- 全量引入大型 SDK 只使用一个函数。
- 同一项目并存多个无必要的大版本。
- 把密钥、Token 和供应商账号写进依赖配置。

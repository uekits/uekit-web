# 组件分层与命名规范

## 1. 品牌和技术前缀

- 产品名：UEKit Web
- npm 包：`@uekits/web`
- Vue 组件前缀：`Ue`，仅在需要品牌区分的公开组件名中使用
- CSS 类前缀：`ue-`
- CSS 变量前缀：`--ue-`
- Composable：`useUe*`
- TypeScript 公开品牌类型：`Ue*`；模块内部类型不强制添加品牌前缀

消费项目本地 import 可以使用自然名称：

```ts
import { Button } from '@/components/ui/button';
```

不要求业务模板写 `<UeButton>`，因为源码已经属于消费方。

## 2. Registry 名称

条目使用 kebab-case：

```text
button
status-tag
pro-table
dashboard-layout
map-amap
charts-echarts
```

名称应表达稳定能力，不带客户、项目或短期版本：

```text
推荐：alarm-summary（如果已泛化为通用告警摘要）
不推荐：tcl-alarm-card、new-table-v2
```

## 3. 目录和文件

```text
registry/<layer>/<item>/
├─ PascalCase.vue
├─ types.ts               # 需要时
├─ composables.ts         # 需要时
└─ index.ts
```

- Vue 文件 PascalCase。
- 普通 TS 文件使用 kebab-case；`index.ts`、`env.d.ts` 等工具约定文件保留固定名称。
- `index.ts` 只做公开导出，不放复杂实现。
- 测试文件使用 `.test.ts`，E2E 使用 `.spec.ts`。

## 4. 分层规则

### Foundation

主题、工具和图标。不能依赖 Pro、Layout、Block 或 Integration。

### UI

基础交互和 Element Plus 适配器。应保持小而稳定。

### Pro

可跨业务使用的企业组合组件，例如查询区、数据表格和详情容器。

### Layout

负责页面结构，通过 Props 和 Slot 接收内容，不请求业务接口。

### Block

更完整的工作区块，可以组合多层组件，但数据访问必须可替换。

### Integration

封装地图、图表、AI 等 SDK 的通用生命周期。

## 5. 依赖方向

允许：

```text
UI → Foundation
Pro → UI / Foundation
Layout → UI / Foundation
Block → Pro / UI / Layout / Foundation
Integration → UI / Foundation
```

禁止反向依赖和循环依赖。

## 6. 业务边界判断

出现以下情况应留在业务项目：

- Props 直接使用某个客户 DTO。
- 组件内部请求具体 `/alarm`、`/device` 接口。
- 文案和字段无法在其他行业成立。
- 组件必须访问业务 Pinia Store 才能运行。
- 权限标识写死为具体系统菜单。

可以先在业务项目验证两次以上，再提取到 Registry。

## 7. 版本命名

Registry 条目使用 SemVer。源码目录不带版本号，版本由 `registry.json` 和 Lock 记录。

不创建：

```text
pro-table-v2/
new-pro-table/
```

不兼容变更应升级条目主版本并提供迁移说明。

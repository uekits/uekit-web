# 注释规范

本文定义 UEKit Web 源码注释和 TSDoc 的最低要求。注释是设计约束的一部分，必须准确、可维护，并能帮助下一位开发者安全修改代码。

## 1. 规则等级

- **MUST**：缺失或错误会影响维护、安全或公开 API 理解，必须处理。
- **SHOULD**：默认需要；代码已经足够直白时可以省略。
- **MAY**：用于复杂模块的额外背景和示例。

## 2. 总原则

### 2.1 解释原因，不翻译代码

注释应回答以下至少一个问题：

- 为什么选择这个实现；
- 哪个协议、兼容性或消费场景形成约束；
- 输入、输出和失败边界是什么；
- 为什么不能使用看起来更直接的方案；
- 修改时最容易破坏什么。

禁止以下注释：

```ts
// 设置 loading 为 true。
loading.value = true;

// 遍历条目。
for (const item of items) {
  // ...
}
```

### 2.2 与代码同步

- 修改实现时 MUST 同步检查附近注释和 TSDoc。
- 已经过期的注释 MUST 删除或修正，不能以“以后再更新”为由保留。
- 注释不得承诺源码没有提供的能力。
- TODO MUST 包含明确任务和可追踪上下文；禁止只写 `TODO: later`。

### 2.3 语言和安全

- 说明文字默认使用简体中文，代码标识、协议字段和命令保持原文。
- 禁止写入真实 Token、密码、API Key、客户名称、内网地址和个人信息。
- 安全检查的注释可以解释威胁模型，但不得给出可利用的真实凭据或内部环境细节。

## 3. 文件头

正式维护的源码文件 MUST 有简洁文件头。文件头描述文件职责和关键边界，不记录作者、创建日期或容易失效的修改历史。

### 3.1 TypeScript、JavaScript 和声明文件

```ts
/**
 * Registry 条目安装流程。
 *
 * @remarks
 * 负责依赖排序、目标路径保护、源码写入和 Lock/Base 更新；不负责解析组件业务语义。
 */
```

适用于：

- `registry/**/*.ts`、`registry/**/*.d.ts`；
- `packages/**/*.ts`；
- `scripts/**/*.{js,mjs,cjs,ts}`；
- 测试源文件；
- 可使用注释语法的工程配置。

仅做公开导出的 `index.ts` MAY 使用单段简短文件头，但不能完全省略。

### 3.2 Vue SFC

Vue 文件头 MUST 是第一个有效内容，并使用 HTML 注释放在 `<template>` 之前：

```vue
<!--
    企业数据表格外壳。

    提供工具栏、分页和加载状态，不负责请求业务数据。
-->
<template>
  <!-- ... -->
</template>
```

文件头 SHOULD 说明组件定位、数据所有权和关键限制，不罗列所有 Props。

### 3.3 CSS 和 SCSS

独立样式文件使用块注释：

```css
/**
 * UEKit 语义 Token 及 Tailwind v4 主题映射。
 */
```

Vue `<style>` 已由 SFC 文件头说明整体职责时，不要求重复样式文件头。

### 3.4 JSON 和生成文件

- 标准 JSON 不支持注释，MUST 保持合法 JSON。
- JSON 字段解释写入对应 Schema 或参考文档，不写伪注释字段。
- `dist/`、Registry 构建 JSON、覆盖率和测试报告等生成产物不补文件头、不人工编辑。
- 第三方许可证和原始 Vendor 文件保持上游内容，不强制添加 UEKit 文件头。

## 4. TSDoc

### 4.1 必须使用的范围

以下声明 MUST 使用 TSDoc：

- Registry 对外导出的 Props、Emits、Slots 和公共类型；
- CLI 导出的函数、接口、选项和协议模型；
- Schema 无法完整表达的协议约束；
- 可能抛错、修改文件、启动进程或访问网络的公共函数；
- 带有重要兼容性、性能或安全边界的内部函数。

以下内容通常不需要 TSDoc：

- 名称和类型已经足够清晰的局部变量；
- 简单事件转发函数；
- 只做一行再导出的内部别名；
- 测试中的普通数据准备步骤。

### 4.2 常用标签

- `@remarks`：设计原因、生命周期、协议或兼容边界。
- `@param`：参数语义、单位、允许范围和特殊值；不重复类型名。
- `@returns`：返回值语义，尤其是空值和所有权。
- `@throws`：可预期错误及触发条件。
- `@typeParam`：泛型承担的角色和约束。
- `@example`：公共 API 的最小可运行用法。
- `@defaultValue`：默认值对使用者有意义时使用。
- `@deprecated`：替代方案和移除计划。
- `@see`：关联协议或仓库文档。

不得为了凑齐标签重复 TypeScript 已能表达的信息。

### 4.3 函数示例

```ts
/**
 * 将 Registry 目标路径解析到消费项目内。
 *
 * @param cwd - 消费项目根目录。
 * @param target - 允许包含 UEKit 公开别名占位符的目标路径。
 * @returns 已解析的绝对路径。
 * @throws 当目标路径越出消费项目根目录时抛出错误。
 */
function resolveTarget(cwd: string, target: string): string {
  // ...
}
```

### 4.4 类型示例

```ts
/** Registry 条目安装选项。 */
export interface InstallOptions {
  /** 消费项目的绝对工作目录。 */
  cwd: string;

  /** 是否允许覆盖未被 Lock 管理的现有文件。 */
  overwrite?: boolean;
}
```

属性语义明显时不必逐字段注释；涉及单位、默认值、安全边界或副作用时 MUST 说明。

## 5. Vue 注释

### 5.1 Props、Emits 和 Slots

- 公共组件的复杂 Props SHOULD 使用命名接口，并说明单位、默认行为和受控/非受控语义。
- Emits SHOULD 说明触发时机；payload 已经直观时不重复字段名称。
- 作用域 Slot SHOULD 说明插槽职责和暴露数据的所有权。
- 简单布尔 Props 不要求逐项写“是否启用”。

### 5.2 computed、watch 和生命周期

- 纯派生 computed 通常不需要注释。
- 产生网络请求、SDK 更新、持久化或跨组件同步的 watch MUST 解释副作用和触发边界。
- 地图、图表、Observer、事件监听和定时器的创建/销毁 MUST 说明资源所有权。
- 异步初始化可能晚于组件卸载时，MUST 说明竞态防护策略。

### 5.3 Template

- 简单布局不写区域翻译注释。
- 多层条件、跨 Slot 协作或无障碍结构不直观时 SHOULD 写短注释。
- 不得在 HTML 注释中放调试信息、业务数据或敏感内容。

## 6. 样式注释

- Token 分组、兼容性 hack 和复杂动画 SHOULD 说明用途。
- `:deep()` 覆盖 Element Plus 时 MUST 说明影响的公开组件、作用域和无法通过公开变量解决的原因。
- 禁止写无边界的“临时覆盖”；临时兼容必须说明移除条件。
- 不为每个选择器、颜色或间距写代码翻译。

## 7. CLI、脚本和安全边界

以下实现 MUST 说明关键原因或威胁边界：

- 目标路径防穿越；
- 本地文件覆盖保护；
- Lock、Base 和 Hash 的关系；
- Registry 循环依赖检测；
- 子进程参数传递和包管理器选择；
- 远端输入校验；
- 临时目录创建和清理；
- 三方合并的保留与冲突策略。

注释不得只写“安全检查”，而要说明防止哪类错误以及失败时采取什么动作。

## 8. 测试注释

- 测试文件头说明被测边界和不覆盖的范围。
- 测试名称已经表达场景时，不重复解释 arrange/act/assert。
- 非显而易见的 fixture、时间控制、网络 Mock 或平台兼容处理 SHOULD 说明原因。
- 回归测试 SHOULD 说明曾经失败的边界，但不得依赖外部不可访问的上下文才能理解。

## 9. 注释校验

仓库使用两类自动检查：

1. ESLint `tsdoc/syntax` 校验已有 TSDoc 的语法；
2. 独立文件头脚本校验正式源码是否缺少文件头。

自动规则不能判断注释是否有价值。代码审查仍 MUST 检查：

- 注释是否与实现一致；
- 是否解释原因、风险或边界；
- 是否暴露敏感信息；
- 是否存在批量生成的废话；
- 修改公开行为时是否同步更新文档。

## 10. AI 编码助手要求

- 新建正式源码时 MUST 按本规范添加文件头。
- 修改复杂逻辑时 MUST 更新相关注释，但不得扩大到无关文件。
- 不得为“提高注释覆盖率”生成逐行翻译。
- 无法确定设计原因时 MUST 根据代码、测试和文档调查；仍不确定时标记待确认，不得编造历史原因。
- 生成的注释与人类编写的注释接受相同检查。

## 11. 审查清单

- 文件头是否说明职责，而不是作者和日期。
- 公共 API、协议、失败和副作用是否有必要 TSDoc。
- 注释是否描述当前行为。
- 资源释放、路径安全和兼容性原因是否清楚。
- 是否存在代码翻译、过期 TODO 或敏感信息。
- Registry 源码中的注释是否对消费项目同样成立。

# 文档编写规范

## 1. 语言

- 默认使用简体中文。
- 命令、代码标识、协议字段保留英文。
- 首次出现 Registry、CLI 等术语时说明含义。
- 避免中英文混杂造成同一概念多种叫法。

## 2. 真实性

- 文档必须对应当前源码。
- 规划内容明确标注“规划中”。
- 不把基础实现描述为完全自动化能力。
- 命令示例应可以复制执行。

## 3. 文档类型

- `guides`：按步骤完成任务。
- `architecture`：解释为什么和如何工作。
- `reference`：精确列出字段、命令和约束。
- `standards`：团队必须遵守的规则。
- `testing`：验证方法和清单。
- `releasing`：发布和回滚操作。

不要把所有内容都堆入 README。

## 4. 标题和结构

- 每个文件只有一个 H1。
- H2/H3 层级连续。
- 长文先给目的和适用对象。
- 操作指南包含前置条件、步骤和验收。
- 危险操作明确说明后果。

## 5. 链接

- 仓库内使用相对链接。
- 不链接构建产物和临时文件。
- 重命名文件时全局更新引用。
- 提交前运行 `pnpm docs:verify`。

## 6. 代码示例

- 标注语言。
- 使用真实命令和字段。
- 不包含真实密钥、Token、账号和内网地址。
- 省略内容时明确用注释表示。

## 7. 同步要求

发生以下变化必须同步文档：

- 新增或修改 CLI 命令。
- 修改 Registry 或配置 Schema。
- 改变安装、覆盖、升级和合并行为。
- 新增条目类型或目录。
- 修改测试和发布流程。
- 改变 Node、Vue、Element Plus 或 Tailwind 基线。

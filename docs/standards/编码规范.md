# 编码规范

本文定义 UEKit Web 的通用编码规则，适用于人类开发者和 AI 编码助手。Vue 组件、样式、依赖和组件分层的专项要求，继续由同目录下的专项规范补充。

## 1. 规则等级

- **MUST**：必须遵守；违反时不能合并。
- **SHOULD**：默认遵守；偏离时必须在代码审查中说明理由。
- **MAY**：按场景选择，但同一模块应保持一致。

规则冲突时，优先级依次为：安全和正确性、公开协议兼容、本文、专项规范、局部习惯。无法判断时先停止扩大改动并发起评审。

## 2. 适用范围和代码画像

不同目录运行在不同环境，不能共用一组未经区分的假设。

| 范围                     | 运行环境               | 主要职责                            | 额外约束                                                      |
| ------------------------ | ---------------------- | ----------------------------------- | ------------------------------------------------------------- |
| `registry/`              | 消费项目浏览器环境     | 可分发 Vue、TypeScript 和样式源码   | 不依赖 UEKit 私有运行时；公开 API 稳定；源码可独立安装        |
| `packages/cli/src/`      | Node.js                | Registry 构建、安装、升级和差异处理 | 路径安全、数据校验、错误信息和跨包管理器兼容优先              |
| `apps/playground/`       | 浏览器                 | 真实消费、展示和视觉验收            | Registry 组件必须通过安装或同步流程进入，禁止形成第二份手工源 |
| `apps/registry-server/`  | Node.js 构建、静态托管 | 发布 Registry 静态文件              | 不承载业务运行时逻辑                                          |
| `scripts/`               | Node.js                | 仓库校验、构建和发布辅助            | 明确输入、输出、副作用和退出码                                |
| `tests/`、`**/*.test.ts` | Node.js 或浏览器       | 单元、集成和端到端验证              | 测试环境按目标代码选择，不默认全部使用 DOM                    |
| 根配置文件               | Node.js 或工具专用     | 工程工具配置                        | 只声明本工具需要的环境和类型                                  |

## 3. 基础原则

### 3.1 小步和聚焦

- MUST 让一次修改只服务于一个明确目标。
- MUST 将纯格式化、类型治理和行为变更拆分，避免审查者无法判断行为是否改变。
- MUST 删除废弃代码，不通过长期注释保留旧实现；历史由 Git 保存。
- SHOULD 优先修改最小必要范围，不顺带重构无关模块。

### 3.2 显式和可读

- MUST 使用可表达领域含义的名称，不使用拼音、无约定缩写或 `data1`、`temp2` 等占位名称。
- MUST 为公开函数声明参数和返回类型。
- SHOULD 使用提前返回降低嵌套；不得用嵌套三元表达复杂分支。
- SHOULD 让函数只承担一个可描述的职责；复杂流程拆成具名步骤。
- 注释要求遵循[注释规范](./注释规范.md)，不得用注释弥补含糊命名。

### 3.3 类型安全

- MUST 开启 TypeScript 严格模式。
- MUST 将外部 JSON、网络响应、环境变量和第三方 SDK 数据视为 `unknown`，校验或收窄后再使用。
- MUST 避免 `any`；确有第三方类型缺口时，将例外限制在适配层并说明原因。
- MUST 处理可能缺失的索引访问和可选值，不使用非空断言掩盖问题。
- SHOULD 优先通过控制流收窄类型，减少 `as` 类型断言。

## 4. 格式

以下是 UEKit Web 的目标格式，由 EditorConfig 和 Prettier 自动执行：

- MUST 使用 UTF-8、LF 和文件末尾换行。
- MUST 使用 4 个空格缩进，不写入 Tab。
- MUST 使用单引号和分号。
- MUST 在多行对象、数组和参数列表中保留尾逗号。
- MUST 将单行宽度控制在 100 个字符内；不可拆分的 URL、命令和类型除外。
- MUST 让 Vue/HTML 多属性标签每行一个属性。
- MUST 让 Markdown 保持人工可读排版，不在保存时自动重排长文。

格式由工具决定。不得为了个人偏好反复手调与格式化器冲突的布局。

## 5. 命名

### 5.1 文件和目录

- 普通目录和 Registry 条目名 MUST 使用 kebab-case。
- Vue 组件文件 MUST 使用 PascalCase，例如 `ProTable.vue`。
- 普通 TypeScript、JavaScript、样式和测试文件 MUST 使用 kebab-case。
- 工具约定文件保留固定名称，例如 `index.ts`、`env.d.ts`、`vite.config.ts`。
- `index.ts` MUST 只维护公开导出，不放复杂实现。

### 5.2 标识符

- 变量、参数和函数 MUST 使用 camelCase。
- 类型、接口、类和 Vue 组件 MUST 使用 PascalCase。
- 布尔值 SHOULD 使用 `is`、`has`、`can`、`should` 等可判断前缀；框架既有 API 除外。
- 函数 SHOULD 使用动词或动宾结构，例如 `resolveTarget`、`validateManifest`。
- 真正的模块级常量 MUST 使用 UPPER_SNAKE_CASE；局部不可变变量仍使用 camelCase。
- `Ue` 类型前缀只用于需要品牌区分的公开类型，不强制用于模块内部类型。

新增事件名称在 TypeScript 声明和模板监听中 MUST 使用 kebab-case；Vue 的 `update:*` 模型事件遵循框架约定。已经发布的 camelCase 事件不得仅为风格统一直接改名，必须先制定兼容、弃用和版本升级方案。

## 6. TypeScript

### 6.1 类型声明

- 对象结构 SHOULD 使用 `interface`；联合、交叉、映射和工具类型 SHOULD 使用 `type`。
- 字符串状态 SHOULD 使用字面量联合或 `as const` 对象；无运行时必要时不使用 `enum`。
- 泛型名称 SHOULD 表达角色，例如 `Row`、`Payload`；简单通用工具允许使用 `T`、`K`、`V`。
- 可复用公开类型 MUST 从条目或模块的公开入口导出。
- `null` 表示明确的空值，`undefined` 表示未提供或尚未产生；同一 API 不得无约定混用。

### 6.2 导入和导出

- MUST 使用 ESM。
- 类型导入 MUST 使用 `import type`，避免产生无意义运行时依赖。
- import 分组顺序为：Node 内置、第三方包、工作区别名、相对路径、副作用样式；组内保持稳定排序。
- Registry 源码 MAY 使用 CLI 能重写的公开别名，但不得依赖消费项目中不存在的私有路径。
- 业务模块和工具模块 SHOULD 使用具名导出；工具要求默认导出的配置文件可使用默认导出。
- 不得创建循环依赖；Registry 依赖方向遵循[组件分层与命名规范](./组件分层与命名规范.md)。

### 6.3 控制流和函数

- MUST 使用 `const`，仅在变量需要重新赋值时使用 `let`；禁止 `var`。
- MUST 为 `if`、`for`、`while` 等控制结构写大括号，即使只有一行。
- MUST 使用 `async`/`await` 表达异步流程，不混用难以追踪的 Promise 链。
- MUST 处理异步失败；禁止空 `catch` 或只记录日志后假装成功。
- SHOULD 将文件系统、网络、进程执行等副作用集中在边界函数中。

### 6.4 类型规则例外

- 禁止 `@ts-ignore` 和 `@ts-nocheck`。
- `@ts-expect-error` 只允许用于验证类型错误或临时第三方兼容，并 MUST 在同一行说明原因和移除条件。
- ESLint 规则禁用必须限制到最小行或文件范围，并说明框架或协议原因。
- 不能仅为通过检查把值强制断言成目标类型。

## 7. Vue 3

Vue 专项 API 设计遵循[Vue 组件开发规范](./Vue组件开发规范.md)，本节只定义通用结构。

### 7.1 SFC 顺序

Vue SFC MUST 使用以下顺序：

```vue
<!-- 文件用途和关键边界。 -->
<template>
  <!-- 模板 -->
</template>

<script setup lang="ts">
// 脚本
</script>

<style scoped>
/* 样式 */
</style>
```

没有模板或样式时可以省略对应区块，但剩余区块的相对顺序不变。

### 7.2 script setup 内部顺序

SHOULD 按以下顺序组织：

1. import；
2. 类型声明；
3. `defineOptions`；
4. `defineProps`、`defineEmits`、`defineSlots`；
5. 常量和响应式状态；
6. computed；
7. watch；
8. 生命周期；
9. 事件处理和业务函数；
10. `defineExpose`。

### 7.3 组件规则

- Props、Emits 和 Slots MUST 使用 TypeScript 声明。
- Props MUST 有明确默认值或可选语义，不得在组件内部修改。
- 模板中的组件名 MUST 使用 PascalCase。
- `v-for` MUST 使用稳定业务键，禁止用数组索引代替可用的稳定标识。
- 模板表达式 MUST 保持简单；复杂计算移入 computed 或具名函数。
- 组件创建的定时器、监听器、Observer、地图或图表实例 MUST 在卸载时释放。
- 样式默认使用 `scoped`；全局样式必须属于主题入口并经过专项评审。

## 8. Element Plus 和第三方 SDK

- Registry 组件 MUST 显式导入使用的 Element Plus 组件和对应样式。
- Registry 源码不得假设消费项目安装了 `unplugin-element-plus` 或执行了 `app.use(ElementPlus)`。
- Playground MUST 保留从 Registry 安装得到的样式导入，不得为减少本地代码而删掉。
- 二次封装 MUST 只依赖第三方库公开 API，不把私有 DOM 当作稳定协议。
- 地图、图表等 SDK MUST 封装加载、更新、销毁和失败路径，不把密钥写入源码。

以上是源码分发架构的明确例外：它不同于团队业务应用中的“插件自动导入样式”，但仍满足按需加载目标。

## 9. 错误、日志和进程输出

- 浏览器源码不得保留调试 `console.log`；必要告警和错误应通过可替换的边界处理。
- CLI MAY 将正常结果写到标准输出，将警告和错误写到标准错误；输出是 CLI 的公开交互，不视为调试日志。
- CLI MUST 让失败命令返回非零退出码，并提供可执行的错误信息，不暴露 Token、密钥或本机敏感路径。
- 脚本 MUST 区分“校验失败”和“工具执行异常”，不得吞掉错误。
- 面向用户的错误信息使用中文时，协议字段、文件名和命令保持原文。

## 10. 测试代码

- 单元测试文件使用 `.test.ts`，Playwright 使用 `.spec.ts`。
- 测试名称 MUST 描述可观察行为和条件，不描述内部实现步骤。
- CLI 和 Registry 测试默认使用 Node 环境；只有真实 DOM 组件测试使用 jsdom 或浏览器环境。
- 测试 MUST 隔离临时目录、端口和环境变量，并在结束时清理。
- 禁止通过降低断言强度、增加任意等待或盲目更新快照修复失败测试。

## 11. 配置文件

- 配置文件只声明自身工具需要的运行环境和 globals。
- Node 配置不得无理由继承 DOM 类型；浏览器配置不得无理由引入完整 Node globals。
- JSON 不支持注释时，不得伪造注释语法；需要解释的配置可在工具兼容的前提下使用 `.js`、`.cjs` 或 `.ts`。
- 不得从参考业务工程机械增加 Tailwind、Router、Pinia 或自动导入插件配置。

## 12. 代码审查清单

- 改动是否只包含声明的目标。
- 公开 API、Registry 协议或 CLI 行为是否意外变化。
- 外部输入是否在使用前校验。
- 异步失败和资源释放是否完整。
- Registry 源码是否仍可在干净消费项目独立运行。
- 是否新增不必要依赖、全局状态或环境假设。
- 注释是否解释原因并与实现一致。
- 测试是否覆盖成功、失败和关键边界。

# 测试体系

## 1. 测试目标

UEKit Web 需要同时验证仓库源码、Registry 发布物、CLI npm 发布物以及调用方实际安装结果。

## 2. 测试金字塔

```text
Playwright 浏览器与视觉交互
          ▲
消费项目构建与 tarball 冒烟
          ▲
CLI / Registry Vitest
          ▲
类型、Lint、样式、格式、文档链接
```

## 3. 静态检查

```bash
pnpm format:check
pnpm lint
pnpm lint:styles
pnpm typecheck
pnpm docs:verify
```

覆盖代码格式、TypeScript、Vue、CSS/SCSS 和 Markdown 内部链接。

`pnpm typecheck` 必须同时出现并通过以下项目：

- `registry typecheck`：直接使用 `vue-tsc` 检查 Registry 事实源；
- `apps/playground typecheck`：检查 CLI 安装后的真实消费副本；
- `packages/cli typecheck`：检查不继承 DOM 类型的 Node CLI。

Registry 和 Playground 是不同边界，两者不能互相替代：前者防止事实源在编辑器中形成推断项目，后者验证别名改写和消费目录中的实际类型行为。

## 4. Registry 验证

```bash
pnpm registry:build
pnpm registry:verify
```

验证条目协议、依赖图、路径、源码哈希以及最终静态站点结构。

```bash
pnpm playground:verify
```

该命令要求 Registry 当前 12 个条目全部安装进 Playground，并逐文件验证 Registry、源码、Base、Lock 版本化来源和 SHA-256 一致。

## 5. 单元测试

```bash
pnpm test
```

重点覆盖：

- Registry 读取和验证。
- CLI 依赖递归解析。
- 路径保护和别名改写。
- 文件覆盖保护。
- Lock 与 Base 生成。
- diff 和 update 行为。
- Dialog/Drawer 事件次数。
- Map/ECharts 动态属性、错误和销毁生命周期。

打包后的 CLI 命令入口另有独立升级冒烟：

```bash
pnpm upgrade:verify
```

该命令验证无修改更新、本地修改保护、无冲突三方合并、冲突旁路文件，以及升级后的 Lock/Base 推进。

## 6. Playground 构建

```bash
pnpm build
```

Playground 是 Registry 组件的真实 Vue 消费环境。构建可以发现源码 import、类型和 Tailwind 扫描问题。

## 7. Playwright

```bash
pnpm exec playwright install chromium
pnpm test:e2e
```

最低覆盖：

- 页面非空并可交互。
- 基础组件和 Pro 组件可见。
- Dialog、Drawer 等关键交互。
- 亮色和深色。
- 桌面和移动端关键宽度。
- 布局没有重叠和溢出。

## 8. npm 发布物冒烟

```bash
pnpm package:check
pnpm smoke:package
```

冒烟测试应：

1. 构建 `@uekits/web`。
2. 执行 `npm pack`。
3. 分别通过 pnpm 和 npm 创建隔离 Vue 项目。
4. 使用 tarball 执行 `init` 和 `add`。
5. 检查源码、配置、Lock、Base 和依赖。
6. 构建消费项目。

这一步能发现 workspace 内测试看不到的发布文件遗漏。

GitHub Actions 在 Node 22 上执行完整质量、pnpm/npm tarball 消费和浏览器 E2E；另用兼容 Node 20 的 pnpm 10 执行 CLI 单测与 tarball 消费。仓库开发仍固定 Node 22 与 pnpm 11，Node 20 是 CLI 消费兼容边界。

## 9. 完整命令

```bash
pnpm check
pnpm test:e2e
pnpm smoke:package
```

`check` 不替代 E2E 和 tarball 冒烟。

## 10. 截图原则

- 固定 viewport、主题和数据。
- 等待字体、动画和异步内容稳定。
- 截图不是唯一断言，还要检查关键 DOM 和交互。
- 失败截图和报告不提交到源码仓库。
- 视觉变化必须由人工确认，不盲目更新基线。

## 11. 真实项目验证

通用组件从真实业务提取或发生重大改动时，至少在一个租户端和一个平台端场景验证，确认没有把单一业务假设带入 Registry。

# 消费项目接入指南

本文面向将 UEKit Web 接入真实业务项目的团队，重点说明组织方式和长期治理。

## 1. 接入前盘点

先确认：

- Vue、Vite、TypeScript 和 Element Plus 版本。
- 是否已经使用 Tailwind CSS。
- 当前全局样式和 Element Plus 变量入口。
- `@` 等路径别名。
- 已存在的同名组件和目录。
- 亮色、深色模式的挂载方式。

建议先在一个非核心页面验证，不要第一次就批量替换全站组件。

## 2. 初始化策略

执行：

```bash
pnpm dlx @uekits/web@latest init
```

检查生成的 `uekit.json`，尤其是：

- Registry 地址。
- `ui`、`pro`、`layouts`、`integrations` 等目录别名。
- 主题 CSS 路径。
- Tailwind 是否启用以及 Preflight 是否关闭。

已有 Tailwind 项目应合并配置，不要并存两个职责重复的 CSS 入口。

## 3. 推荐接入顺序

```text
theme
  ↓
icon / avatar / status-tag / button
  ↓
search-panel / pro-table
  ↓
detail-drawer / form-dialog
  ↓
dashboard-layout
  ↓
map-amap / charts-echarts
```

先建立主题和基础组件，再迁移高级组件。这样问题更容易定位。

## 4. 本地封装边界

安装后的源码可以修改，但建议分三类管理：

1. **直接采用**：不修改上游组件。
2. **受控定制**：在本地组件中修改，同时保留上游追踪。
3. **业务封装**：在业务目录再包装一层，不修改通用组件。

如果差异只属于一个业务模块，优先使用业务封装；如果所有项目都需要同样修改，应回馈 UEKit Registry。

## 5. 禁止混用的方式

业务页面应逐步统一引用本地 UEKit 组件：

```ts
import { Button } from '@/components/ui/button';
```

不建议同一语义在页面中随机混用：

```vue
<el-button />
<Button />
<LegacyButton />
```

但无需为了形式统一而包装 Element Plus 的每一个组件。UEKit 应优先封装有设计约束、交互约束或组合价值的能力。

## 6. Git 管理

必须提交：

- 安装后的源码。
- `uekit.json`。
- `uekit.lock.json`。
- `.uekit/bases/`。
- package.json 和锁文件变化。

推荐一个组件安装或升级对应一个独立提交，便于回滚。

## 7. 团队升级流程

```text
创建升级分支
  ↓
升级 @uekits/web CLI 或使用指定版本
  ↓
pnpm dlx @uekits/web@0.1.2 diff <item>
  ↓
pnpm dlx @uekits/web@0.1.2 update <item> [--merge]
  ↓
人工处理 .uekit-merge
  ↓
类型检查、单测、构建、Playwright
  ↓
代码评审后合并
```

不要在所有开发人员本地自动无提示升级组件源码。

## 8. 多项目治理

多个业务项目复用 UEKit 时应建立：

- 统一的推荐 CLI 版本。
- 允许安装的 Registry 环境。
- 每季度或每个发布周期的升级窗口。
- 本地修改回馈上游的评审流程。
- 设计 Token 变更公告。
- 兼容性测试项目清单。

## 9. 从传统组件库迁移

迁移不应一次性重写。推荐：

1. 先接入主题，审查 UEKit Base Layer 对 `html`、`body`、字体、背景和 `box-sizing` 的全局影响。
2. 选用户管理等典型页面建立验证链路。
3. 提取 Avatar、StatusTag 等低风险组件。
4. 再迁移 Table、Drawer、Dialog 等共享结构。
5. 删除确认不再引用的本地重复实现。
6. 使用 Playwright 对比迁移前后。

## 10. 验收标准

- 页面不直接依赖 UEKit 远端运行时代码。
- 实际只安装所需组件和依赖。
- 亮色、深色、桌面和移动端正常。
- 本地修改能被 `diff` 识别。
- 重装不会覆盖未追踪文件或已修改文件。
- 业务项目可以独立构建和部署。

# 新增组件指南

本文以新增一个 Registry 组件为例，说明从判断归属到发布验收的完整流程。

## 1. 先判断是否应该进入 UEKit

满足以下条件时适合进入 Registry：

- 能在多个项目或多个业务模块复用。
- API 不依赖具体行业实体。
- 视觉和交互可以由语义 Token 表达。
- 能提供稳定的 Props、事件和插槽。
- 有明确的空态、加载态、禁用态和错误态。

以下内容通常不应进入：

- 固定的项目、设备、告警字段。
- 直接依赖某个客户接口的数据请求。
- 仅为单个页面临时拼接的布局。
- 无法脱离业务 Store 运行的组件。

## 2. 确定分层

| 类型                   | 适用内容              | 常见目标目录            |
| ---------------------- | --------------------- | ----------------------- |
| `registry:foundation`  | 主题、工具、图标      | `src/lib`、`src/styles` |
| `registry:ui`          | 基础 UI 适配组件      | `src/components/ui`     |
| `registry:pro`         | 企业高级组合组件      | `src/components/pro`    |
| `registry:layout`      | 页面和应用布局        | `src/layouts`           |
| `registry:block`       | 可组合业务区块        | `src/blocks`            |
| `registry:integration` | 地图、图表、AI 等集成 | `src/integrations`      |

完整约定见[组件分层与命名规范](../standards/组件分层与命名规范.md)。

## 3. 创建标准源码

以 `empty-state` 为例：

```text
registry/ui/empty-state/
├─ EmptyState.vue
├─ index.ts
└─ types.ts
```

组件源码要求：

- 使用 TypeScript 和 `<script setup>`。
- Props、Emits 和公开类型可导出。
- 不读取业务 Store。
- 不硬编码项目别名以外的消费项目目录。
- 使用 `--ue-*` 语义 Token 或 Tailwind 语义类。
- Element Plus 组件和样式按需导入。

## 4. 登记 Registry 条目

在根目录 `registry.json` 添加条目：

```json
{
  "name": "empty-state",
  "type": "registry:ui",
  "version": "0.1.0",
    "description": "用于列表和详情区域的统一空状态。",
    "url": "items/empty-state/0.1.0.json",
    "dependencies": ["element-plus@^2.14.0"],
    "devDependencies": [],
    "registryDependencies": ["theme", "icon"],
    "files": [
        {
            "path": "registry/ui/empty-state/EmptyState.vue",
            "target": "{ui}/empty-state/EmptyState.vue"
        },
        {
            "path": "registry/ui/empty-state/index.ts",
            "target": "{ui}/empty-state/index.ts"
        }
  ]
}
```

注意：

- `url` 必须严格使用 `items/<name>/<version>.json`。
- `dependencies` 是消费项目运行时 npm 依赖，必须包含明确版本或范围。
- `devDependencies` 是消费项目构建和开发依赖。
- `registryDependencies` 是其他 Registry 条目名称。
- `path` 相对仓库根目录。
- `target` 使用 `uekit.json` Alias 占位符描述消费项目路径。

## 5. 构建和校验 Registry

```bash
pnpm registry:release
pnpm registry:build
pnpm registry:verify
pnpm playground:verify
```

`registry:release` 会把新版本写入 `registry-releases/items/<name>/<version>.json`。如果同名同版本已经归档但内容不同，命令会拒绝覆盖；此时必须先按 SemVer 提升条目版本。

构建会检查：

- JSON Schema。
- 名称和类型。
- 缺失或循环的 Registry 依赖。
- 源文件是否存在。
- 路径穿越和非法绝对路径。
- 重复源路径和重复目标路径。
- 文件内容和 SHA-256。

不要通过直接修改构建产物或 `registry-releases` 中已经公开的条目 JSON 解决错误。

## 6. 添加 Playground 示例

示例需要覆盖：

- 默认状态。
- 关键变体和尺寸。
- 加载、禁用、空态或错误态。
- 长文本和窄容器。
- 亮色和深色主题。

示例文案应表达真实用法，不写“这是一个组件”“点击这里测试”等内部说明。

## 7. 添加测试

至少包含：

- Registry 构建测试。
- 组件关键 Props 和事件的 Vitest 测试。
- 影响视觉或交互时补充 Playwright。
- 安装后路径和 import 可解析的冒烟测试。

## 8. 用临时项目验证安装

启动本地 Registry 后，在干净项目执行：

```bash
pnpm dlx /path/to/uekits-web.tgz init
pnpm dlx /path/to/uekits-web.tgz add empty-state
pnpm typecheck
pnpm build
```

同时检查：

- npm 依赖是否自动安装。
- 别名是否正确改写。
- `uekit.lock.json` 是否记录文件。
- `.uekit/bases` 是否保存基准。
- 再次添加时不会无提示覆盖本地修改。

## 9. 文档和版本

- 在文档中添加用途、API 和示例。
- 按 SemVer 更新 Registry 条目版本，并执行 `pnpm registry:release` 生成不可变归档。
- 对公开 CLI 包行为有影响时添加 Changeset。
- 按[发布前验收清单](../testing/发布前验收清单.md)完成验收。
